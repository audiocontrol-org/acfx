<LessonPart id="concept" kicker="Concept" title="What a state-variable filter is" active>

A **state-variable filter** (SVF) is a single second-order structure that
produces a **lowpass**, **bandpass**, and **highpass** output *at the same
time*, from the same internal state, on every sample. You do not pick one and
rebuild the filter to get another — all three come from one topology. Two
controls shape them: **cutoff frequency**, the corner where the filter turns
over, and **resonance** (Q), how much the response peaks at that corner.

Because it gives three responses from one structure with two simple controls,
the SVF is a good filter to start with. It introduces the two ideas that later,
more complex filters reuse: a coefficient that sets frequency and a coefficient
that sets resonance, feeding back through a pair of integrators.

As an analog prototype, the three outputs are three numerators over **one shared
denominator** — same poles, different zeros. The shared denominator is the key:
cutoff and resonance move the *poles*, and all three outputs respond together.

<MathWell label="Transfer functions" note="one denominator · three numerators">
<pre>{`H_LP(s) = ω₀²          / (s² + (ω₀/Q)·s + ω₀²)
H_BP(s) = (ω₀/Q)·s     / (s² + (ω₀/Q)·s + ω₀²)
H_HP(s) = s²           / (s² + (ω₀/Q)·s + ω₀²)`}</pre>
</MathWell>

In discrete time the SVF is a small feedback network of two integrators. The
signal flow: the input, minus the lowpass and a resonance-scaled bandpass, forms
the **highpass**; the highpass is integrated to make the **bandpass**; the
bandpass is integrated again to make the **lowpass** — and those two feed back
into the next sample. One coefficient `f` sets the integrator gain (cutoff), one
coefficient `q` sets the damping (resonance).

<MathWell label="Discrete recurrence" note="Chamberlin form · per sample">
<pre>{`HP[n] = x[n] − LP[n−1] − q·BP[n−1]
BP[n] = f·HP[n] + BP[n−1]
LP[n] = f·BP[n] + LP[n−1]

f = 2·sin(π·fc/fs)          q ← resonance`}</pre>
</MathWell>

Lower `q` narrows the peak (higher Q); as `q` approaches zero the corner rings
harder, until the filter self-oscillates. The next two sections let you hear
this, then see the pole responsible move toward the unit circle.

</LessonPart>

<LessonPart id="observe-it" kicker="Observe it" title="See the filter respond">

The same filter, shown three ways as you change its controls. The **frequency
response** shows what each frequency is multiplied by; the **z-plane poles and
zeros** show where the resonance comes from; and the **impulse response** shows
how the filter rings in time.

Sweep cutoff and the whole response slides along the frequency axis. Raise
resonance and the **pole moves toward the unit circle** — the same moment the
response grows its peak and the impulse rings longer. This is the transfer
function from the first section made visible: the pole's radius corresponds to
the resonance.

<LessonPart id="build-it" kicker="Build it" title="How it's built in code">

The filter is built up in four layers, each a real file in the repository. A
**lab** works out the math and decides how to implement it; a **primitive**
wraps that into a small, reusable filter; an **effect** exposes it with
parameters; and an **adapter** compiles it for a target — here, the browser.
Each layer below shows the code and a way to run it.

<CheckpointLadder />

One property holds across all four layers: the core filter code knows nothing
about the browser, or any platform. Dependencies point inward — each adapter
reaches into the core, never the other way — so the same filter code compiles
unchanged whether the target is a plugin, hardware, or the web.

</LessonPart>
